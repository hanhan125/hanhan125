摘  要
传统课堂教学管理中，签到流程耗时长、学生注意力状态难以实时量化评估，这两个问题长期存在。班级人数达到40人及以上时，点名式签到通常需要5分钟，且难以杜绝代签、补签等学术不端行为。教师对学生专注度的判断主要依赖个人经验，缺乏客观数据支撑。针对上述问题，本文基于人脸检测与注意力分析技术，设计并实现了一套教学智慧管理辅助装置，重点完成了动态签到模块的开发工作。
系统采用前后端分离的B/S架构。后端以Python FastAPI框架构建RESTful API服务，通过WebSocket协议实现服务端到客户端的实时数据推送，采用SQLite轻量级嵌入式数据库进行数据持久化存储。前端基于React框架结合TypeScript语言进行组件化开发，以Vite作为构建工具，借助ECharts可视化图表库呈现专注度时序分析曲线。人脸检测与特征分析环节，选用Google MediaPipe Face Landmarker模型对摄像头采集的视频流进行逐帧分析，实时提取468个3D面部关键点。在此基础上分别计算眼睛纵横比、嘴巴纵横比、头部姿态偏移量和头部运动速度四项特征指标，并通过多维度加权融合算法（权重依次为0.42、0.33、0.25）生成0至100分区间的综合专注度评分。
系统核心功能主要包括三部分。其一，动态签到—，首次人脸检测即触发签到，摄像头识别到学生面部的瞬间自动完成出勤记录。其二，专注度分析，从表情、头部姿态和行为稳定性三个维度量化学生课堂参与状态，为教师提供数据驱动的教学辅助参考。其三，课堂实时看板前端页面通过WebSocket长连接接收后端推送的签到事件与注意力采样数据，以表格列表和ECharts折线图的形式动态呈现全班学生的实时状态。经功能测试验证，各模块在标准PC环境下运行稳定，人脸检测帧率达到15至30FPS，WebSocket消息推送延迟控制在100毫秒以内。系统无需依赖专用硬件设备，可在普通计算机上独立部署运行。


关键词：人脸检测；动态签到；专注度分析；MediaPipe；FastAPI；WebSocket
Abstract
In the traditional classroom teaching management, the check-in process takes a long time and the student 's attention state is difficult to be quantitatively evaluated in real time. These two problems exist for a long time. When the number of students in a class reaches 40 or more, it usually takes 5 minutes to sign in by name, and it is difficult to prevent academic misconduct such as substitutes and supplements. Teachers ' judgment of students ' concentration mainly depends on personal experience and lacks objective data support. In view of the above problems, this paper designs and implements a set of teaching wisdom management auxiliary device based on face detection and attention analysis technology, focusing on the development of dynamic sign-in module. 
The system adopts a B / S architecture with front-end and back-end separation. The back-end uses the Python FastAPI framework to build RESTful API services, implements real-time data push from the server to the client through the WebSocket protocol, and uses the SQLite lightweight embedded database for data persistence storage. The front end is based on the React framework combined with the TypeScript language for component development. Vite is used as a construction tool, and the ECharts visual chart library is used to present the concentration time series analysis curve. In the process of face detection and feature analysis, Google MediaPipe Face Landmarker model is used to analyze the video stream collected by the camera frame by frame, and 468 3D facial key points are extracted in real time. On this basis, the four characteristic indexes of eye aspect ratio, mouth aspect ratio, head posture offset and head movement speed are calculated respectively, and the comprehensive attention score between 0 and 100 points is generated by multi-dimensional weighted fusion algorithm ( the weights are 0.42,0.33 and 0.25 respectively ). 
The core function of the system mainly includes three parts. First, the dynamic check-in-the first face detection triggers the check-in, and the moment the camera recognizes the student 's face automatically completes the attendance record. Secondly, focus analysis, which quantifies students ' classroom participation status from three dimensions of expression, head posture and behavioral stability, provides teachers with data-driven teaching auxiliary reference. Third, the front-end page of the classroom real-time Kanban receives the check-in event and attention sampling data pushed by the back-end through the WebSocket long connection, and dynamically presents the real-time status of the whole class in the form of a table list and an ECharts line chart. After functional testing and verification, each module runs stably in the standard PC environment, the face detection frame rate reaches 15 to 30 FPS, and the WebSocket message push delay is controlled within 100 milliseconds. The system does not need to rely on dedicated hardware devices and can be deployed independently on an ordinary computer.

Keywords: Face Detection; Dynamic Sign-in; Attention Analysis; MediaPipe; FastAPI; WebSocket

1  绪论
1.1  设计目的
课堂是学校教育的核心阵地，教学管理效率与质量直接影响人才培养的实际效果。传统课堂教学中，签到通常依赖教师逐一点名或者学生手动登记两种方式，二者均存在明显不足。从时间成本来看，班级人数超过40人的时候，教师逐一点名并勾画点名册通常需要5分钟甚至更长。以每周4学时、每学期16周计算，一个学期中仅点名环节就消耗约160分钟的课堂教学时间。并且从数据准确性来看，人工签到难以解决代签、补签和漏签等行为，签到记录的真实性和完整性缺乏可靠保障[1]。
上述诸多问题在教育信息化加快推进的形势下变得越发突出。2018年的时候，教育部发布了《教育信息化2.0行动计划》，清晰地提出要推动信息技术跟教育教学实现深度融合，促使课堂教学的智能化水平得到提高。[2]。然而当前市面上比较成熟的智慧教室解决方案，比如说依靠RFID射频识别或者蓝牙信标的考勤系统，一般都得部署专用的硬件设备，单单一间教室的改造成本能够达到数万元甚至还要更高。如此高昂的建设投入致使智慧教室在中小规模教学场所的普及进程比较缓慢。
近年来深度学习技术在计算机视觉领域取得了突破性进展。Google于2020年开源的MediaPipe框架，将高精度的3D人脸关键点检测能力集成进轻量级神经网络中，使得在普通CPU计算设备上实现实时人脸分析成为切实可行的技术方案[3]。与此同时，WebSocket协议（RFC 6455）的标准化和React等现代前端框架的普及，为构建低延迟、高交互性的浏览器端实时数据看板奠定了技术基础[4][5]。
基于上述背景，本毕业设计打算实现一套基于人脸检测与注意力分析的教学辅助管理装置，重点完成其中的动态签到模块。系统会用一个普通的USB摄像头来采集教室画面，然后借助MediaPipe Face Landmarker模型，实时检测人脸并提取468个3D关键点。在此基础上，系统会进一步计算EAR、MAR、头部偏移和头部运动速度等指标，再通过一个多维度加权融合模型，算出综合的专注度评分。与此同时，系统把人脸检测成功的那一刻作为签到触发条件，自动完成动态签到记录。整个系统只需在一台普通PC电脑上运行即可，不用特意部署昂贵的硬件设备。
1.2  本文研究的实用性
本研究的实用性集中体现在以下四个层面。
第一，显著提升课堂签到效率。动态签到功能可在不打断正常教学节奏的前提下，通过摄像头检测人脸，在识别到学生面部的瞬间自动完成签到记录。原本需要3至5分钟的课堂点名，可缩短到几秒钟内完成，教师能将节省出的时间直接投入到教学当中。与传统RFID刷卡或手机蓝牙签到相比，本方案无需学生携带额外设备，也不需排队依次操作，签到过程对师生几乎没有感知。[6]。
第二，传统课堂中，教师对学生注意力的判断主要依赖个人经验，有经验的教师可通过学生的神态、坐姿和视线方向大致判断专注程度，但这种方式较为主观，且难以覆盖全班所有学生，也无法形成连续的量化记录。本系统通过提取面部关键特征点，持续计算EAR、MAR和头部姿态角度等指标，为每位学生生成0到100分的综合专注度评分序列，为教师提供数据驱动、连续量化的课堂参与度参考。教师可依据专注度曲线识别教学过程中的低谷期与高参与度期，进而调整教学策略。[7]。
第三，部署成本极低。系统运行于普通Windows电脑上，只需配备一个USB摄像头，而这类摄像头在绝大多数教学计算机上已是标配。软件方面全部采用开源技术栈：FastAPI（MIT许可）、React（MIT许可）、MediaPipe（Apache 2.0许可）、SQLite（公共领域），无需支付任何商业软件许可费用。系统不依赖云服务器，免去了服务器租赁和带宽成本，日常维护只需具备常规的信息技术管理能力即可。这种零硬件增量成本、零软件许可成本、零云端运维成本的部署方式，使得经费有限的学校也能够使用智能化的课堂管理工具。第四，前后端分离架构为系统带来了良好的可扩展性。后端通过标准化RESTful API对外提供全部服务，前端作为独立应用运行。未来如需开发微信小程序或移动端App作为新的客户端，只需按照相同的API规范开发对应前端界面，后端服务无需修改即可复用。这一设计有效保护了当前的开发投入，也降低了系统后续升级演进的技术门槛。[5]。

1.4  相关技术研究
1.4.1  人脸检测与特征点提取技术
人脸检测是计算机视觉领域的经典问题之一，其核心任务是在图像或视频帧中准确定位人脸的位置和范围。在人脸检测技术的发展脉络中，2001年由Viola和Jones提出的基于Haar-like特征和级联AdaBoost分类器的检测框架具有里程碑意义。该算法在当时的计算条件下首次实现了实时人脸检测，在工程实践中得到广泛应用[8]。然而，基于手工设计特征的检测方法对光照条件变化、面部姿态偏转和局部遮挡的鲁棒性存在天然局限，在教室这种光照不均、学生姿态多样的真实场景中，单纯依靠Viola-Jones框架难以满足高可靠性的检测需求。
随着深度卷积神经网络（CNN）的兴起，人脸检测的性能获得了质的提升。Zhang等人于2016年提出的MTCNN（Multi-task Cascaded Convolutional Networks）模型设计了三级级联网络架构（P-Net、R-Net、O-Net），将人脸检测与人脸关键点定位两项任务进行联合优化，在检测精度与推理速度之间取得了较好的平衡，已成为人脸检测领域被广泛引用的基线方法之一[9]。在关键点检测方向，Dlib库提供的68点人脸关键点检测方案因其接口简洁、使用方便而被大量学术研究和工程项目采纳，但其关键点数量有限，仅覆盖面部轮廓和主要器官的粗略位置，在需要精细分析面部微表情的场景中信息量不足。洪文亮等人对人脸关键点检测技术进行了系统综述，梳理了从经典统计形状模型（ASM/AAM）到深度热图回归方法的完整技术演进脉络[10]。
Google于2020年开源的MediaPipe框架为人脸特征分析提供了全新的技术方案。Lugaresi等人在其论文中详细阐述了MediaPipe的模块化计算图设计理念：开发者通过组合已有的感知计算组件构建推理流水线，框架负责管理跨平台（移动端、桌面端、边缘设备）的硬件加速和内存优化[3]。MediaPipe Face Mesh模块能够实时估计468个3D面部关键点的空间坐标，覆盖了面部轮廓、眉毛、眼睛、鼻子、嘴巴等全部面部区域，关键点密度是传统68点模型的近7倍。2023年，Google将Face Mesh升级为Face Landmarker任务API，在检测精度和推理速度方面均有进一步提升，且提供了更简洁的Python调用接口。本系统选用MediaPipe Face Landmarker作为核心视觉分析引擎，主要基于三点考虑：关键点密度高（468点对注意力细粒度分析的信息充足性）、推理实时性强（CPU模式下可达15-30FPS）、跨平台部署便利。
1.4.2  课堂场景人脸分析研究现状
近些年来，不少研究者都把人脸分析技术运用到课堂教学场景之中，此方向已然成为一个颇为活跃的研究范畴。2016年，Soukupova和Cech提出了一种依据面部关键点的实时眨眼检测算法，这个算法定义了一个名为眼睛纵横比（EAR）的既简洁又高效的量化指标。该指标是通过去计算眼睛垂直方向关键点距离之和与水平方向距离的比值，进而实时展现眼睛的开合状况[11]。该研究进行的实验显示，正常睁眼状态的时候EAR值大概在0.25至0.35这个区间，眯眼时EAR值会随着眼缝缩小而逐渐减小，完全闭合的时候就趋近于零。这一指标给后续诸多课堂注意力分析研究提供了具有基础性的测量工具。
邱红、刘石坚等人从时频域联合建模的角度提出了一种分层图卷积专注度识别方法。该方法将学生面部的动态时序特征与频域特征进行融合，利用图卷积网络对面部关键点之间的空间拓扑关系进行建模，在课堂专注度分类任务上取得了较高的识别准确率[12]。邵帅、李卓等人针对远程课堂场景，提出了结合面部姿态角度检测模块与身体动作行为识别模块的双通道框架，通过将面部朝向与身体动作两类信息进行互补融合，对学生的听课专注度进行综合评估[7]。这些研究工作表明，基于面部关键点的注意力分析在技术上是可行的，在应用上是具有实际价值的。
1.4.3  实时Web通信与数据可视化技术
在B/S架构的实时系统中，服务端与客户端之间高效、低延迟的数据传输是实现实时功能的核心前提。2011年，IETF将WebSocket协议标准化为RFC 6455[4]。与HTTP的请求-响应模式不同，WebSocket在客户端与服务端之间建立一条持久的全双工TCP连接，双方在任意时刻均可主动发送数据。在连接保持期间，WebSocket的数据帧头部开销仅为2至14字节，远低于HTTP请求动辄数百字节的头部开销，这使其天然适合需要高频、低延迟消息推送的实时应用场景。在Python生态体系中，FastAPI框架通过底层Starlette库提供了对WebSocket的原生支持，开发者仅需使用Python的async/await协程语法即可实现高效的WebSocket服务端逻辑，同时无需额外集成就能获得FastAPI自动生成的交互式API文档（Swagger UI）和基于Pydantic的请求/响应数据校验功能[13]。
在前端数据可视化方面，百度前端团队做出了ECharts这个库，后来还把它贡献给了Apache软件基金会。Li Deqing等人在《Visual Informatics》期刊上发布了一篇论文，这篇论文里详细讲了ECharts的声明式可视化设计思路，也就是开发者只要写JSON格式的配置项，就能快速搭出带各种交互功能的可视化图表[14]。ECharts库在大数据量渲染场景下进行了深度优化，所以它能够流畅展示数万个数据点的时序曲线。React作为当前前端主流框架，该框架通过虚拟DOM机制和组件化开发思想有效提升了复杂用户界面的代码组织和可维护性[5]。TypeScript相当于给JavaScript加了一套静态类型，在代码编译阶段就能抓出一大堆类型相关的错误，这样一来，前端项目在反复改来改去的时候，重构的风险就能降低不少。

2.1  功能需求
2.1.1  系统整体功能需求
教学智慧管理辅助装置作为一个面向课堂教学场景的完整软件系统，该系统需要满足以下六个方面的整体功能需求。第一，教室与学生管理功能。该系统应支持教室基本信息的创建与查询，具体信息包括教室名称和创建时间；同时该系统还应支持学生信息的录入与管理，基础字段包括学号、姓名等，教师应能通过界面操作批量导入或动态添加学生记录。第二，课堂场次管理功能。该系统应支持创建课堂场次（Session），并且每个场次需要记录每次课堂教学的开始时间和结束时间；每个场次还须关联到特定教室，系统应在场次存续期间持续收集签到数据和注意力采样数据。第三，动态签到功能。当摄像头检测到学生面部时，该系统会自动触发签到记录，说白了就是学生什么都不用做就能完成签到，签到数据会实时推送到前端看板，系统还支持按课堂场次查询签到状态与历史记录。第四，专注度分析功能。该系统基于人脸关键点检测结果，从表情、头部姿态和行为稳定性三个维度综合计算学生的课堂专注度评分（0至100分），并且系统会将评分结果实时推送至前端，同时持久化存储到数据库中。第五，实时数据看板功能。该系统以列表形式展示全班学生的签到状态与专注度评分数据，同时以折线图形式呈现选定学生的专注度随时间变化的趋势曲线，所有数据的更新延迟应控制在500毫秒以内。第六，局域网访问功能。同一局域网内的其他设备（比如教师的手机或平板电脑）应能通过浏览器访问前端看板页面，并且系统需要妥善处理跨域资源共享（CORS）的相关安全配置。
2.1.2  个人模块需求——动态签到模块的详细功能界定
动态签到模块是本人在团队任务中的核心负责部分，负责团队任务中的动态签到功能，以及技术要求中的人脸检测和活体检测内容。本节对动态签到模块的功能需求进行详细界定和说明。动态签到模块的核心设计理念是检测即签到，学生在课堂环境中无需执行任何主动签到操作，签到动作由人脸检测事件自动触发。当学生进入USB摄像头视野、面部被MediaPipe Face Landmarker模型成功检测并定位时，系统自动记录该学生的签到时间和签到状态（标记为已到）。这一机制的根本优势在于解耦了签到与学生主动行为之间的绑定关系。传统签到模式中，学生错过了点名时机即被标记为迟到或缺勤；而动态签到模式下，只要学生在课堂期间的任意时刻进入摄像头视野，签到即可完成。摄像头采集进程以15至30FPS的帧率持续运行，当Face Landmarker首次成功检测到人脸关键点时，采集模块立即向签到上报接口发送签到记录请求。后端验证课堂场次ID和学生ID合法性后，在attendance表中写入status为present的签到记录。
后端生成签到记录后，立即通过WebSocket将attendance事件广播至对应教室频道的所有客户端。然后前端接收到事件，通过React状态管理更新签到标签显示。所有的签到记录都存储在SQLite数据库的attendance表中。系统提供按课堂场次查询最新签到状态的API接口，支持前端在页面初始化或WebSocket重连后通过HTTP轮询获取全量状态。考虑可能没有摄像头的演示场景，本系统还额外支持演示数据模式。教师点击一键生成演示数据按钮后，后端自动创建示例教室和4名演示学生，按预设时间间隔持续推送模拟数据[15]。
2.2  经济效益与社会贡献分析
经济效益层面，本系统最突出的特征是零增量硬件成本。系统运行所需的唯一外部设备为USB摄像头，这在绝大多数教学计算机上属于已有配置。与市场上单教室动辄上万甚至数万元的智慧教室整体解决方案相比，本系统的硬件投入几乎为零。软件层面，全部技术栈均为开源软件（FastAPI、React、MediaPipe、SQLite、ECharts），不涉及任何商业许可费用。以每堂课节省3分钟点名时间为保守估算，教师每周授课10节、每学期16周计算，每学期可节省约480分钟的课堂时间，这部分时间可直接转化为教学内容的产出，其经济价值是真实可量化的。
社会贡献层面，本系统的价值主要体现在促进教育公平和推动数据驱动教学两个维度。教育公平方面，本系统因其极低的部署门槛，使得硬件经费有限的中小规模学校和农村地区教学点同样有能力引入智能化课堂管理工具，有助于缩小不同区域、不同层次学校之间在教育信息化水平上的客观差距。《中华人民共和国个人信息保护法》（2021年施行）对生物识别信息的处理提出了严格的法律要求。本系统的设计对此给予了高度重视：摄像头采集的原始视频流仅用于本地实时分析，不进行持久化存储和网络传输；提取的面部特征值（EAR、MAR、头部角度等）为脱敏后的聚合统计量，不可逆向还原为原始人脸图像；所有数据存储于本地SQLite文件，不存在自动向云端或第三方上传的行为。这些设计选择确保了系统在提供课堂智能化服务的同时，严格遵循了个人信息保护的法律规范。

2.4  毕业课题与非技术因素的关联分析
2.4.1  与法律、法规因素关联分析
本系统进行设计与实现时要全面考量我国个人信息保护、数据安全方面法律法规提出的要求。2021年开始施行的《中华人民共和国个人信息保护法》作出了明确规定，相关主体处理人脸信息这类生物识别信息时要获得个人的单独同意，而且相关主体需采取必要的安全保护措施。本系统在设计的时候非常看重个人信息保护，具体通过以下设计决策得以体现：本系统运行的视频流仅仅是用于实时人脸检测、特征点提取，系统不会对原始视频数据进行持久化存储也不会向外传输，本系统的摄像头采集分析模块只在本地内存当中进行图像处理，该模块提取的面部特征值属于脱敏后的量化统计值，这些数值不能逆向还原成原始人脸图像。本系统在本地计算机上运行，全部数据都存于本地的SQLite数据库文件内，不存在自动化向云端或者第三方服务器上传数据的情况。这些设计决策保证了本系统在具备智能化教学辅助功能时，能够严格依照个人信息保护的法律规定。
《中华人民共和国网络安全法》、《中华人民共和国数据安全法》对网络信息系统的安全运行与数据安全管理给出了明确要求。本系统用的是本地部署架构，此架构不会对外暴露网络服务端口，而且本系统默认不连接互联网，这种设计有效避免了数据远程泄露、网络攻击的安全风险。本系统启动的时候只绑定本地回环地址，系统的局域网功能要用户主动配置IP和防火墙规则之后才可以启用，这种机制展现了最小化暴露的安全设计原则。
2.4.2  与健康、安全、环境因素关联分析
在健康因素方面，本系统通过摄像头进行人脸分析的机制属于被动式、非侵入性的监测方式，学生无需佩戴任何穿戴式传感器或进行特殊操作，避免了因设备佩戴不适或操作不便引发的健康问题。摄像头以常规帧率（15至30FPS）工作，不存在高强度频闪或辐射等潜在健康风险。系统的注意力分析结果仅作为教学参考数据呈现给教师，不会直接向学生显示评判性信息或进行公开排名，避免因注意力“分数化”可能引发的心理压力和同伴比较等负面心理影响。
在安全因素方面，系统的使用场景为教室环境，摄像头通常安装于讲台区域或教师计算机上，不存在因设备安装引发的人身安全隐患。系统的软硬件架构简单可靠，无高功耗设备或特殊供电需求，符合教室环境的电气安全标准。系统在运行过程中不产生电磁干扰或其他影响教学设备正常工作的因素。
在环境因素方面，本系统全部以软件形态部署于现有教学计算机，不需要额外的硬件制造和运输环节，碳排放低，符合绿色发展和可持续设计的理念。系统代码量适中，运行时计算资源消耗少，不会显著增加计算机的能耗，对环境的影响微乎其微。
3  概要设计
3.1  系统整体设计
3.1.1  技术方案选型对比分析
在概要设计阶段，设计者对后端框架、数据库引擎、人脸检测模型和前端技术栈四个方向分别进行了候选方案的对比论证，以下逐一阐述选型逻辑。后端框架选择方面，候选方案包括Django、Flask和FastAPI。Django是Python生态中功能最全面的全栈Web框架，内置ORM（Django ORM）、管理后台（Django Admin）、表单系统等完善组件，但其重量级的设计理念与本系统轻量化本地部署的定位存在错配；Flask以微框架著称，灵活简洁，但当需要支持异步处理（asyncio）、WebSocket、自动API文档生成和数据校验等功能时，均需额外选择和集成第三方扩展，增加了开发过程中的决策成本和集成调试工作量。FastAPI结合了三个方面的优势：性能层面，基于Starlette异步框架和Uvicorn服务器，其并发处理能力接近Node.js和Go语言级别；开发效率层面，框架自动生成Swagger UI交互式API文档，内置基于Pydantic的请求/响应数据校验，利用Python类型提示提供代码编辑器智能感知；生态层面，与Python asyncio异步编程生态无缝集成，WebSocket支持作为一等公民原生内嵌[13]。经综合比较，确定以FastAPI作为后端开发框架。
数据库选型方面，候选方案包括MySQL、PostgreSQL和SQLite。MySQL和PostgreSQL是成熟的企业级关系型数据库管理系统，在并发访问控制、复杂查询优化和运维监控方面功能强大，但其部署要求运行独立的数据库服务进程（mysqld或postgres），需要配置用户和权限，与单机一键部署的系统定位存在明显矛盾。SQLite是一个嵌入式的关系型数据库引擎，将全部数据库对象（表、索引、视图、触发器等）存储在操作系统上的单一文件中。它无需独立服务进程，无需任何安装配置操作，支持标准SQL语法和ACID事务特性。通过SQLModel库（基于SQLAlchemy核心和Pydantic的ORM层），可在Python中以声明式模型定义的方式完成数据库表结构定义和CRUD操作，代码可读性高、类型安全性好。对于本系统单用户访问、数据量适中、无并发写入冲突的数据使用特征，SQLite的适用性是最为匹配的[17]。
人脸检测引擎选型方面，综合比较了OpenCV Haar Cascade、Dlib HOG+SVM、MTCNN和MediaPipe Face Landmarker四个方案。OpenCV Haar Cascade的帧处理速度最快，但检测精度有限，对侧脸、仰俯角和遮挡的鲁棒性不足[8]。Dlib HOG方案在正面人脸检测上效果良好但同样对角度变化敏感。MTCNN在检测精度上有明显提升，但其三级级联网络结构在CPU上推理时计算开销较大[9]。MediaPipe Face Landmarker采用针对实时推理场景优化的轻量级神经网络架构，配合内部的GPU加速管线，在普通集成显卡的笔记本上即可实现15至30FPS的实时推理帧率，同时输出468个3D面部关键点，无论是关键点数量还是空间维度（3D而非2D），都远超Dlib 68点和MTCNN 5点方案。综合检测精度、推理速度和部署便利性三个维度的权衡结果，确定MediaPipe Face Landmarker为核心视觉分析引擎。
前端技术栈选型方面，比较了纯HTML/CSS/JS、Vue.js和React三种方案。纯方案开发效率低、代码组织结构差，不适合中等规模以上的前端项目。Vue.js学习曲线相对平缓、上手速度快，但其生态中TypeScript支持的成熟度和第三方组件库的丰富程度与React相比仍有差距。React依托Facebook/Meta持续投入和全球最活跃的前端开发者社区，在TypeScript集成成熟度、第三方组件生态规模和性能优化工具链方面居于领先地位。其声明式组件模型和单向数据流机制与Vite构建工具和ECharts图表库的React封装组件兼容性良好[5][14]。最终确定React + TypeScript + Vite + ECharts的前端技术组合。
3.1.2  系统组成与架构
本系统由五个核心组成部分构成。后端API服务：基于FastAPI框架，负责提供RESTful API接口（共计18个端点，涵盖教室管理、学生管理、课堂场次管理、签到管理、注意力数据管理、摄像头控制和演示数据管理等业务领域），处理业务逻辑，执行数据持久化操作，并通过WebSocket Hub实现教室频道的实时事件广播。摄像头采集模块：以独立子进程方式运行，与后端主服务进程解耦，负责打开本地USB摄像头、调用MediaPipe Face Landmarker模型逐帧检测人脸关键点、计算EAR/MAR/头部姿态等特征指标、通过加权融合算法生成专注度评分，以及周期性地通过HTTP POST请求将注意力数据和签到数据上报至后端API。前端Web看板（frontend/src/App.tsx）：基于React + TypeScript实现，通过WebSocket长连接接收后端实时推送的签到事件和注意力采样数据，通过2秒间隔HTTP轮询器作为降级补充，以表格列表和ECharts折线图分别展示学生实时状态和专注度历史趋势。SQLite数据库（tas.db）：以单文件形式存储于后端项目根目录，存储教室、学生、课堂场次、签到记录和注意力样本五类结构化数据。WebSocket Hub：管理各教室频道的WebSocket连接集合，提供带互斥锁保护的广播方法[18]。
系统各组成部分之间的数据流关系为：教师通过前端界面触发开始上课操作后，前端向POST /api/camera/start接口发送HTTP请求；后端创建课堂场次记录后，通过subprocess.Popen以独立子进程方式启动摄像头采集脚本，同时传递API基地址、教室ID、学生ID和课堂场次ID等运行参数；采集进程逐帧进行人脸检测，在检测到人脸后计算多维特征指标并生成专注度评分，通过POST /api/attention和POST /api/attendance接口将数据上报至后端；后端将数据持久化写入SQLite后，通过WebSocket Hub将数据封装为JSON事件消息广播至对应教室的所有已连接前端客户端；前端接收到事件消息后解析并更新React状态，触发页面重新渲染，系统总体架构如图3-1所示。

3.1.3  系统业务流程
系统的完整业务流程涵盖从启动到退出的全部操作步骤。第一步，教师双击项目根目录的启动.bat文件，脚本自动检测Python和Node.js运行环境是否就绪，根据需要安装缺失的依赖包，然后依次启动后端Uvicorn服务器（绑定0.0.0.0:8001，支持局域网访问）和前端Vite开发服务器（绑定0.0.0.0:5173，配置--host参数），获取本机局域网IPv4地址后打印至控制台供移动设备访问参考，最后调用操作系统默认浏览器打开http://127.0.0.1:5173看板页面。第二步，教师在Web界面选择目标教室（首次使用时系统自动创建名为示范教室A的默认教室和4名示例学生），或手动创建新教室和添加学生。第三步，教师点击开始上课（打开摄像头）按钮，后端依次完成停止上一轮采集进程、创建新课堂场次记录、启动摄像头采集子进程的操作。第四步，摄像头采集进程持续循环执行帧读取→人脸检测→特征计算→评分融合→数据上报的处理流程，前端看板同步实时更新签到标签和专注度得分。第五步，教师在需要时可点击学生列表中任意学生行，下方专注度曲线卡片随即通过ECharts展示该学生自课堂开始以来的专注度得分变化时序曲线。第六步，教学结束时教师点击下课/结束按钮，后端终止摄像头采集子进程、清理僵尸进程、记录课堂场次的结束时间，并通过WebSocket广播session:ended事件，系统总体业务流程如图3-2所示。

4.1  概念结构设计
4.1.1  实体识别与属性分析
通过对第2章需求分析结果的系统梳理，从业务领域中识别出五个核心实体。教室（Classroom）实体代表教学活动进行的物理空间单元，属性包括教室ID（主键，自增整型）、教室名称（字符串类型，长度1至64字符）和创建时间（日期时间类型，默认值为UTC当前时间戳）。学生（Student）实体代表参与课堂教学活动的个体，属性包括学生ID（主键，自增整型）、学号（字符串类型，长度1至32字符，建有B树索引以支持按学号快速检索）和姓名（字符串类型，长度1至64字符）。课堂场次（SessionRecord）实体代表一次具体课堂教学活动的记录，属性包括场次ID（主键，自增整型）、关联教室ID（外键，整型，建有索引以支持按教室查询课堂记录）、场次标题（字符串类型，长度1至128字符）、开始时间（日期时间类型，默认UTC当前时间戳）和结束时间（日期时间类型，可空，仅在教师执行下课操作后才被赋值）。签到记录（Attendance）实体代表某位学生在某个课堂场次中的签到事件，属性包括签到ID（主键，自增整型）、关联场次ID（外键，整型，建有索引）、关联学生ID（外键，整型，建有索引）、签到状态（字符串类型，取值限制为present/late/absent三种，建有索引，默认值为present）和签到时间戳（日期时间类型，建有索引以支持按时间排序查询）。注意力样本（AttentionSample）实体代表某位学生在某个课堂场次中某采样时刻的综合注意力评估数据，包含12个属性：样本ID（主键）、关联场次ID（外键，建有索引）、关联学生ID（外键，建有索引）、综合专注度得分score_attention（浮点类型，取值范围0至100，建有索引以支持按得分范围排序和筛选）、表情维度得分score_expression（浮点类型）、头部姿态维度得分score_headpose（浮点类型）、行为维度得分score_behavior（浮点类型），以及眼睛纵横比EAR、嘴巴纵横比MAR、偏航角yaw、俯仰角pitch和翻滚角roll共5个可空浮点型原始证据字段（用于支持更细粒度的后续分析），采样时间戳（日期时间类型，建有索引）。
4.1.2  实体关系分析
五个实体间的关系能够归纳成下面五组。教室和课堂场次有一对多的包括关系，就是说一间教室能够容纳发生在不一样时间段的多个课堂场次，不过每个课堂场次只能在唯一确定的教室当中进行。课堂场次与签到记录存在一对多的包括关系，也就是一个课堂场次包括多名学生的签到事件进而产生多条签到记录，然而每条签到记录肯定源自并且只源自一个特定的课堂场次。课堂场次与注意力样本有着一对多的包括关系，即一个课堂场次在它的持续时间里对多名学生多个采样时刻的数据进行采集，这个过程会产生多条注意力样本记录，而且每条注意力样本记录归属于唯一的课堂场次。学生与签到记录有一对多的参与关系，即一名学生能够参与多个课堂场次并且在每个场次中都有一份签到记录，但是每条签到记录只属于一名学生。学生与注意力样本具有一对多的采样关系，即一名学生在同一课堂场次的不同时刻会被系统多次采样，这个过程会产生多条注意力样本记录。数据库整体的E-R关系如图4-1所示。

4.2  逻辑结构设计
根据概念结构设计的结果，将五个实体分别转化为以下关系模式（数据库表结构），主键以下划线标识，外键字段以括号注明引用关系。

4.3  物理结构设计
在物理数据库层面，本系统选用SQLite 3作为数据库引擎。SQLite数据库在物理上表现为操作系统上的单一二进制文件，数据库文件在系统首次启动时由FastAPI应用启动事件处理器（on_event(“startup”)）调用SQLModel.metadata.create_all()方法自动创建，因此无需用户手动执行任何建库建表的DDL操作[17]。SQLite的嵌入式存储特性与本系统本地单机部署的场景高度契合：数据库文件与应用程序同目录放置，数据备份和迁移仅需复制单个文件；无需独立的数据库守护进程，降低了系统资源的总体占用量和进程管理复杂度；底层使用B树索引结构，对主键查询和外键等值查询均能提供常量时间的查找性能。在索引策略方面，各表的核心查询字段均已建立索引，常见的查询模式，按教室查课堂记录、按课堂查签到列表、按学生查历史注意力样本，均能利用对应索引实现高效检索。注意力历史查询等高频操作通过SQL的LIMIT子句限制单次返回的记录数量上限（默认300条，参数范围10至2000条），避免因历史采样数据随时间累积增长而影响查询性能。数据写入方面，所有INSERT操作均通过FastAPI依赖注入管理的独立数据库会话（Session）完成，异常发生时自动回滚以维护数据的完整性和一致性。

5.1.1  摄像头启动接口设计与实现
动态签到模块在后端的核心入口是摄像头启动接口（POST /api/camera/start）。该接口的处理流程包含以下四个顺序执行的步骤。首先来清理一下环境，通过调用camera_runner.stop() 方法，去终止前一次或许还在运行的摄像头采集子进程，利用PowerShell命令（Get-Process python | Where-Object {...}）查找并且强制终止系统里可能残留的realtime_camera.py僵尸进程，以此保证后续启动不会受到历史进程状态的干扰。接着是上下文数据自动准备，查看请求体中的classroom_id参数，如果没有提供或者对应的教室不存在，那就调用数据库查询，在classroom表中查找或者自动创建一间默认教室，也就是命名为示范教室A，同样的，如果请求里没有提供students_id_list参数，就在student表中自动创建4名演示学生，他们的student_no分别是2024001至2024004，姓名分别为张三、李四、王五和赵六。接着创建课堂场次并进行初始记录，于sessionrecord表中插入一条新记录，将当前UTC时间作为started_at字段的值。随后在attendance表中，为该场次下的每一名学生分别插入一条status为absent的初始签到记录，同时通过WebSocket广播场次启动（session:started）事件。最后是启动摄像头采集子进程，调用Python标准库subprocess.Popen方法，以命令行参数的形式向realtime_camera.py脚本传递API基地址（http://127.0.0.1:8001）、教室ID、学生ID列表（以逗号分隔）、课堂场次ID、摄像头设备索引号（默认为0），并设置creationflags=subprocess.CREATE_NEW_CONSOLE标志，让采集进程在新弹出的控制台窗口中运行，窗口里以OpenCV的cv2.imshow方法实时显示摄像头画面（叠加人脸检测框和专注度得分文字标注），方便教师直接观察检测效果。
5.1.2  摄像头采集核心算法实现
摄像头采集模块realtime_camera.py是动态签到功能在算法层面的核心实现所在，以下对其完整处理流程做逐环节展开阐述。
初始化阶段。脚本启动后首先检查backend/models/目录下是否存在face_landmarker.task模型文件；若文件不存在，则通过httpx库从Google Storage的MediaPipe模型托管地址自动下载至本地。模型加载成功后，调用vision.FaceLandmarker.create_from_options()方法创建推理实例，配置参数包括：base_options=model_asset_path（指定模型文件路径）、running_mode=VisionRunningMode.VIDEO（使用视频流推理模式以利用帧间时序信息）、num_faces=1（当前版本配置为单人脸模式，以优化单帧推理耗时）。同时，通过cv2.VideoCapture(camera_index)打开指定索引的USB摄像头，获取视频流句柄。
主循环处理。进入无限循环后，每轮迭代执行以下子步骤。（a）帧读取：调用cap.read()方法从摄像头读取一帧BGR格式图像，记录当前时间戳作为帧时间戳（毫秒精度），提供给Face Landmarker的detect_for_video()方法用于内部帧间追踪。（b）格式转换与推理：将OpenCV输出的BGR三通道图像通过cv2.cvtColor转换为RGB格式，调用mp.Image(image_format=mp.ImageFormat.SRGB, data=rgb_image)封装为MediaPipe Image对象，传入landmarker.detect_for_video(mp_image, frame_timestamp_ms)执行推理，返回FaceLandmarkerResult对象。（c）结果判断：检查result.face_landmarks列表是否非空。若为空表示当前帧未检测到任何有效人脸，则在OpenCV画面上通过cv2.putText绘制“NO FACE”提示文字，跳过后续特征计算步骤，直接进入帧显示环节。（d）关键点提取：若检测成功，从result.face_landmarks[0]（第一张人脸的关键点列表）中按MediaPipe Face Mesh标准索引号提取注意力分析所需的关键点子集。左眼6个关键点：索引[33, 160, 158, 133, 153, 144]，分别对应左眼左角（33）、左上眼睑（160、158）、左眼右角（133）、左下眼睑（153、144）；右眼6个关键点：索引[362, 385, 387, 263, 373, 380]，对应的几何位置对称于左眼；嘴巴4个关键点：上唇中心点索引13、左侧嘴角索引78、下唇中心点索引14、右侧嘴角索引308；鼻尖关键点：索引1，用于评估头部在画面中的空间位置和帧间运动速度。（e）特征指标计算：基于上述关键点的二维归一化坐标（x、y值在0至1区间，乘以图像宽高后转换为像素坐标），依次计算四项特征指标。
眼睛纵横比的计算属于专注度评估里很关键的部分。2016年Soukupova和Cech最先提出了这个指标，其定义是：假设眼睛的六个关键点坐标从P1到P6，这里面P1和P4分别是眼睛水平方向的两个角点，P2和P6 、P3和P5分别是上下眼睑的两对垂直对应点，那么眼睛纵横比的计算公式是：
EAR = (‖P2 - P6‖ + ‖P3 - P5‖) / (2 × ‖P1 - P4‖) （公式5-1）。
该公式的物理意义是，它表示眼睛垂直方向打开程度的加权平均值与水平方向宽度的比值。当眼睛处于正常睁开状态时，在这个公式里，分子也就是上下眼睑垂直距离之和，大概是在0.08至0.12（归一化坐标单位）这个范围，而分母也就是水平宽度，大致处于0.25至0.35范围，所以EAR值通常是落在0.25至0.35之间。当眼睛闭合时，上下眼睑就快要重合了，分子接近于零，EAR值也跟着下降到0.18以下。在代码实现过程中，系统会分别针对左右两眼独立计算EAR值，之后取这两者的算术平均值当作当前帧的眼睛状态表征。然后，该系统通过线性映射函数把EAR数值转变为0至100分的眼睛状态评分，具体的转换方法是：系统设定[0.18, 0.32]为正常范围，当EAR低于0.18时系统输出评分为0（代表完全闭合），当EAR高于0.32时系统输出评分为100（代表完全睁开），当EAR处于这个区间的时候系统采用线性插值进行映射。这个阈值的确定参考了Soukupova论文里的实验统计数据、本系统在实际运行中的观察调优[11]。
嘴巴纵横比MAR的计算采用与EAR平行的计算逻辑：系统利用上唇中心点与下唇中心点之间的归一化垂直距离作为分子，利用左侧嘴角与右侧嘴角之间的归一化水平距离作为分母，然后计算两者的比值。当学生安静听讲、嘴巴闭合或微张时，MAR值通常低于0.05；当学生打哈欠、大声说话或咀嚼时，这些行为会显著增大嘴巴的垂直开合幅度，MAR值可达0.1以上。系统在将MAR映射为表情得分时采用分段阈值策略：当MAR小于0.05时系统给出95分（表示闭合或正常微表情状态），当MAR处于0.05至0.09区间时系统给出80分，当MAR处于0.09至0.14区间时系统给出55分，当MAR大于0.14时系统给出30分。头姿中心度得分反映的是学生头部（以鼻尖位置代表）相对于摄像头画面几何中心的偏移程度。将鼻尖的归一化坐标（xn, yn）与画面中心点（0.5, 0.5）计算欧氏距离偏移量，偏移量为0时（学生正对摄像头且头部居中）得满分100，偏移量增大时得分按线性衰减至0（衰减系数设为2.2，通过实验调优确定，以确保得分对偏移具有适当的敏感度而又不因轻微晃动导致得分剧烈波动）。行为稳定性得分通过跟踪鼻尖关键点在连续帧之间归一化坐标位置的变化来度量。计算相邻两帧鼻尖位置的欧氏位移后，使用指数移动平均（EMA，平滑系数α=0.85）对位移序列进行平滑处理，滤除检测噪声引入的高频抖动。EMA平滑后的运动速度根据以下分段阈值映射为稳定性得分：速度低于0.002时得92分（头部基本保持静止，高度专注），0.002至0.006得75分，0.006至0.012得55分，高于0.012得35分（频繁的大幅度头部晃动，专注度低）。
加权融合评分是注意力评估的最终输出环节。综合专注度得分score_attention由三个维度得分按以下权重组合计算：S_attention = 0.42 × S_expression + 0.33 × S_headpose + 0.25 × S_behavior (算法5-1).权重分配的逻辑依据如下：表情维度（以EAR为核心）占据最高权重0.42，原因是眼睛的睁开程度是反映个体清醒状态和注意力水平最直接、最不易受意识控制的生理信号—，人处于疲惫或注意力涣散状态时，眼睑下垂、眨眼频率变化是难以主观掩饰的客观生理现象；头部姿态维度权重0.33，补充反映学生对教学内容的空间关注方向，面向讲台和屏幕时头姿得分高，长时间扭头或低头时头姿得分下降；行为稳定性维度权重0.25，虽然头部晃动确实与注意力波动存在相关关系，但该维度受个体习惯差异（部分学生存在不自觉的抖腿、转笔等不影响注意力的微动作）和检测噪声的影响较大，因而赋予相对较低的权重。

5.3  课堂实时看板模块
课堂实时看板模块是系统的前端展示核心，实现于frontend/src/App.tsx组件（476行React + TypeScript代码）。该模块通过合理的状态管理架构和数据同步策略，将后端推送的签到和注意力数据高效地转化为直观的网页交互界面。
状态管理层面，App.tsx使用React的useState钩子管理10余个核心状态变量，包括：WebSocket连接状态status、错误信息error、教室列表classrooms、学生列表students、当前选中教室ID classroomId、当前课堂场次ID sessionId、签到状态字典attendance（Record<number, AttendancePayload>类型）、最新注意力数据字典latestAttention、当前选中学生ID selectedStudentId、专注度历史数据数组history（限定最近600个数据点以实现内存可控的滚动窗口）、摄像头绑定学生ID cameraBoundStudentId和摄像头运行标志cameraRunning。此外还使用useRef钩子创建selectedStudentIdRef和sessionIdRef两个可变引用，使WebSocket消息回调函数在其闭包中能够访问最新的选中学生ID和场次ID值，这是React函数式组件中解决闭包过期值问题的标准模式。
WebSocket通信管理通过connectWs()函数实现。该函数在classroomId变化时由useEffect钩子自动触发：首先关闭已存在的旧WebSocket连接实例，然后创建指向ws://127.0.0.1:8001/ws/classrooms/{classroomId}端点的新WebSocket对象，注册四个事件回调，onopen回调设置status为connected，onclose和onerror回调设置status为disconnected，onmessage回调是核心处理逻辑：使用JSON.parse解析事件数据，根据event.type字段进行分发。若为attendance类型，直接将payload中的签到数据以student_id为键写入attendance状态字典。若为attention类型，更新latestAttention状态字典，同时判断该事件的学生ID与当前选中的学生ID是否一致：若一致则将该数据追加至history数组（如果当前history长度已达600则移除最旧的一个数据点）。若为session类型，根据event字段值（started/ended）更新sessionId状态。
双通道数据同步策略是本模块在可靠性方面的关键设计决策。主要数据通道为WebSocket实时推送，负责传输签到事件、注意力采样和课堂场次状态变更等实时性要求高的数据。辅助数据通道为HTTP定时轮询，通过2秒间隔（setInterval定时器）分别调用GET /api/attendance/latest和GET /api/attention/latest接口获取全量最新数据，该通道的意义在于为WebSocket可能发生的消息丢失（网络瞬时断开后重连期间的gap）提供降级补充。两个通道共同作用，保证前端展示的签到状态和注意力数据与实际服务器端状态的最大差值不超过2秒（轮询间隔）。专注度历史曲线数据不通过WebSocket实时推送（避免高频数据导致前端数组暴涨），而是在selectedStudentId或sessionId状态变化时，通过单次GET /api/attention/history请求按需拉取全量历史数据。ECharts图表渲染通过echarts-for-react组件实现，配置文件通过useMemo在history数组变化时自动重新计算：x轴为时间戳格式化的本地时间字符串，y轴为score_attention值（保留一位小数），启用smooth曲线平滑渲染、tooltip交互提示和connectNulls断点自动连接功能[14]。课堂实时看板界面效果如图5-5所示。

为量化验证专注度评分模型在区分不同课堂行为状态方面的有效性，设计了一个包含四种典型课堂行为场景的对比测试。每种场景下，同一名测试者模拟对应的课堂行为并持续约15分钟，期间系统以每秒4次的频率记录各项注意力指标。测试结束后，计算各场景下的综合专注度得分均值（保留至整数）和三个子维度得分的算术平均值。四种场景及结果如下：正常听讲场景，学生面向屏幕、眼睛平视、坐姿端正，综合得分88分（表情91、头姿93、行为79），EAR均值约0.28，MAR均值约0.03，头部偏移量小且运动速度低。低头看手机场景，学生头部低垂面向手机屏幕，综合得分42分（表情56、头姿28、行为45），EAR因视角不佳无法稳定计算故均值偏低，而头部俯仰角（pitch）大幅偏离正常范围直接导致头姿得分骤降。打哈欠犯困场景，学生表现出明显的张嘴哈欠和眼睑下垂特征，综合得分35分（表情22、头姿68、行为51），MAR均值达0.18显著高于其他场景，EAR均值降至0.13。交头接耳场景，学生频繁转动头部与邻座交流，综合得分50分（表情63、头姿42、行为39），行为稳定性得分因连续大幅头部转向而明显下降。

7  开发环境和系统部署
7.1  开发环境配置
本系统采用前后端分离的B/S架构，在Windows 11专业版（64位）操作系统下完成开发。
（一）后端开发环境
后端使用Python语言开发，通过Python内置的venv模块创建独立虚拟环境来管理项目依赖。后端核心依赖包包括：FastAPI作为Web框架、Uvicorn作为ASGI服务器、SQLModel作为ORM工具（底层基于SQLAlchemy和Pydantic）、Pydantic-Settings用于配置管理、HTTPX用于异步HTTP请求、WebSockets用于WebSocket实时通信、Python-Multipart用于文件上传支持。此外，摄像头采集和图像处理部分通过可选的requirements-camera.txt额外安装OpenCV-Python、MediaPipe和Pillow。
后端启动命令如下：在backend目录下创建虚拟环境并安装依赖后，通过Uvicorn启动FastAPI应用，监听所有网络接口以支持局域网访问：
cd backend
python -m venv .venv
.venv\Scripts\pip install -r requirements.txt
.venv\Scripts\python -m uvicorn app.main:app --host 0.0.0.0 --port 8001
后端配置通过Pydantic-Settings从.env文件加载，主要配置项包括：应用名称、SQLite数据库路径、CORS跨域白名单（允许localhost:5173/5174及局域网IP段192.168.0.0/16和10.0.0.0/8的跨域请求）。系统首次启动时自动初始化数据库并创建默认教室。后端运行配置图如图7-1所示。

图7-1后端运行配置图
（二）前端开发环境
前端采用React框架，使用TypeScript编写，以Vite作为构建工具。前端核心依赖包括：react和react-dom（UI框架）、echarts和echarts-for-react（版本^3.0.6）用于数据可视化图表展示。开发依赖包括：@vitejs/plugin-react用于React的Vite插件支持、ESLint系列用于代码规范检查、@types/react和@types/react-dom提供TypeScript类型定义。Node.js版本要求为18.0.0及以上。

7.2  系统部署方案
本系统面向授课教师的本地化便捷部署需求，采用前后端分离的运行架构，部署流程简洁明了，具体部署步骤说明如下。
第一步，环境准备。 确保目标计算机已安装Python 3.10或更高版本（安装时勾选"Add Python to PATH"选项以自动配置环境变量）和Node.js 18或更高版本。验证方法为分别在命令行中执行python --version和node --version确认可执行文件正确安装。此外，如需使用摄像头实时检测功能，需确认计算机配备USB摄像头且在Windows相机应用中可正常获取画面。
第二步，项目文件部署。 将项目根目录完整拷贝至目标计算机的任意路径下，建议路径不含中文字符和空格以避免潜在的编码或路径解析问题。项目目录结构分为backend（后端代码）和frontend（前端代码）两个独立子目录。
第三步，后端部署。 进入backend目录，使用Python内置venv模块创建独立虚拟环境并安装依赖（cd backend、python -m venv .venv、.venv\Scripts\pip install -r requirements.txt）。如需使用真实摄像头进行人脸检测和注意力分析，还需额外安装摄像头相关依赖（.venv\Scripts\pip install -r requirements-camera.txt）依赖安装完成后，启动Uvicorn服务器。为支持同一局域网内的移动设备访问，需绑定0.0.0.0地址：.venv\Scripts\python -m uvicorn app.main:app --host 0.0.0.0 --port 8001。后端首次启动时，系统会自动创建SQLite数据库，初始化Classroom、Student、SessionRecord、Attendance和AttentionSample五张数据表，并自动建立一个名为"默认教室"的初始教室记录。后端提供的主要接口包括：/api/classrooms（教室管理）、/api/students（学生管理）、/api/sessions（课堂场次管理）、/api/attendance（签到记录管理）、/api/attention（专注度数据上报），以及/ws/classrooms/{classroom_id}（WebSocket实时数据推送）。CORS中间件已配置允许localhost、127.0.0.1以及192.168.0.0/16和10.0.0.0/8两个局域网IP段的跨域请求，确保移动设备可正常调用后端API。
第四步，前端部署。 进入frontend目录，安装Node.js依赖并启动Vite开发服务器：（cd frontend、npm install、npm run dev）Vite开发服务器默认监听5173端口，并通过内置代理将/api请求转发至后端http://127.0.0.1:8001，将/ws的WebSocket请求转发至ws://127.0.0.1:8001，从而在开发阶段避免跨域问题。启动后在浏览器中访问http://127.0.0.1:5173即可打开系统看板页面。
第五步，局域网移动设备访问。 教师在同一WiFi网络下的手机或平板等移动设备上访问系统时，需先获取本机局域网IPv4地址（可通过命令行执行ipconfig查看）。在frontend目录下创建.env.development.local文件，参考.env.lan.example的格式填写本机IP地址（VITE_API_BASE=http://192.168.x.x:8001、VITE_WS_BASE=ws://192.168.x.x:8001）。配置完成后重新执行npm run dev，在移动设备浏览器地址栏输入http://192.168.x.x:5173即可访问系统前端页面。由于后端CORS中间件已配置允许局域网IP段的跨域请求，移动设备可正常加载前端页面并调用后端API。首次访问时Windows防火墙可能弹出网络访问权限提示窗口，点击“允许访问”即可。
第六步，停止系统。 关闭浏览器页面后，分别在运行Uvicorn和Vite的两个终端窗口中按Ctrl+C组合键，依次终止后端服务器和前端开发服务器的运行进程。
补充部署说明。 本方案面向本地开发和课堂演示场景设计，操作简便、无需额外配置。若未来需要将该系统部署为正式生产环境以支持多班级并发使用，建议进行以下调整：将React前端执行npm run build构建为静态HTML/CSS/JS文件，通过Nginx进行反向代理和静态资源服务；后端使用Gunicorn（Linux环境）或多Worker模式的Uvicorn启动以提升并发请求的处理能力；将SQLite替换为MySQL或PostgreSQL等支持多连接并发写入的关系型数据库；根据实际网络环境配置防火墙规则、HTTPS证书和用户认证机制。




结  论
本毕业设计针对传统课堂教学管理中签到效率低下、学生注意力状态难以量化评估等现实问题，设计并实现了一套以人脸检测与注意力分析为核心技术路线的教学智慧管理辅助装置，重点完成了其中动态签到模块的开发工作。全文的工作内容与取得的主要成果可归纳为以下六个方面。
第一，完成了系统的需求分析与整体架构设计。通过对传统课堂签到和注意力监测场景的深入剖析，清晰地界定了系统的6项功能需求和4项非功能需求，特别对个人负责的动态签到模块的功能需求进行了逐条详细分解。经过对后端框架、数据库引擎、人脸检测模型和前端技术栈四个核心方向的候选方案对比论证，确定了以FastAPI + SQLite + MediaPipe Face Landmarker + React为核心的技术架构方案，并采用前后端分离的B/S架构风格进行设计。
第二，设计并实现了动态签到模块。该模块采纳检测即签到的无感机制作为核心设计理念，学生无需执行任何主动签到操作，摄像头在捕捉到学生面部的瞬间自动触发签到记录生成。模块采用三层架构实现：数据采集层由OpenCV + MediaPipe驱动，以15至30FPS的帧率持续进行视频流分析；业务逻辑层执行签到合法性校验、数据持久化和WebSocket事件广播；数据展示层通过React组件实现签到状态的双通道（WebSocket推送+HTTP轮询降级）同步更新。动态签到机制从根本上改变了传统课堂签到的操作范式，将原本耗时3至5分钟的点名过程压缩至数秒内完成。
第三，设计并实现了专注度分析模块。从MediaPipe Face Landmarker输出的468个3D面部关键点中，按照解剖学功能分组原则选取与注意力分析直接相关的一组关键点子集，分别计算四项特征指标，眼睛纵横比EAR、嘴巴纵横比MAR、头部中心偏移度和帧间运动速度。四项指标通过各自的分段或线性映射函数转化为0至100分的维度得分后，以0.42/0.33/0.25的经验优化权重进行加权融合，输出综合专注度评分。这个四层评分模型（特征提取→指标计算→维度评分→融合输出）能够从表情、头姿和行为三个维度综合刻画学生的课堂注意力状态。第六章的对比实验数据验证了评分模型在区分正常听讲、低头看手机、打哈欠犯困和交头接耳四种典型课堂行为方面的有效性。
第四，设计并实现了课堂实时看板模块。前端采用React + TypeScript + Vite + ECharts的现代Web技术栈进行组件化开发，通过WebSocket长连接接收后端的实时数据推送，以表格列表和ECharts折线图两种可视化方式分别呈现学生的实时状态快照和专注度历史演进趋势。看板的数据同步采用WebSocket实时推送为主、HTTP定时轮询为辅的双通道策略，确保了在正常网络条件和瞬时网络波动情况下数据展示的连续性和可靠性。前端页面的CSS采用自定义属性管理主题变量和响应式网格布局，在PC浏览器和移动设备屏幕上均可获得可用的浏览体验。
第五，完成了系统的集成测试和功能验证。设计了覆盖主要功能场景和边界条件的12个测试用例，对动态签到、专注度分析和课堂实时看板三个核心功能模块进行了独立测试和端到端系统集成测试。全部测试用例验证通过，测试结果表明系统功能完整、核心逻辑正确、运行稳定。系统的关键性能指标，人脸检测帧率15至30 FPS、WebSocket消息延迟低于100毫秒、前端首屏加载耗时1至2秒，达到了预期的设计和用户体验要求。
第六，设计了一套完整的一键本地化部署方案。部署流程仅需双击启动.bat一个操作动作，脚本自动完成环境检查、依赖安装、双服务启动和浏览器打开的完整序列，启动成功后教师即可立即开始使用系统功能。这一方案的设计目标是将部署的技术门槛降至最低限度，使缺乏专业技术背景的普通教师也能够独立完成系统的安装和启动。
在取得上述成果的同时，本系统在以下方面仍存在改进和完善的空间。首先，当前系统的人脸到学生身份的映射采用基于画面横向位置顺序的简化方案，在严格的多学生识别场景下存在因学生更换座位而导致数据绑定错误的可能。未来的改进方向是引入专门的人脸识别模型，通过提取高维面部特征向量进行1:N的身份匹配，从根本上确保身份映射的准确性和鲁棒性。其次，当前版本的注意力分析以单人检测模式（num_faces=1）运行为主，尚未覆盖真实课堂中多学生同时面对摄像头的复杂场景。未来的演进方向是切换至多人脸检测模式，并引入人脸跨帧跟踪（Face Tracking）算法为每张检测到的人脸分配唯一ID并维持时间一致性，使系统能够在真正意义上的多人同框教室环境中发挥作用。最后，系统目前尚未具备对历史签到和注意力数据的聚合分析和统计报表功能，教师可以查看单次课堂的实时数据，但无法获取跨多次课堂的长周期趋势分析（如某位学生在本学期各周的专注度对比、全班的出勤率月度变化等）。引入数据分析和统计报表模块，将分散的实时数据点聚合为具有教学管理决策价值的趋势信息和规律发现，是系统从数据采集工具向教学决策辅助工具升级的关键一步。
